# 2026-03-18 - 工作日志

## 主要工作：GEM 论文数据合成 pipeline 实现

### 1. 论文分析（arxiv.org/pdf/2601.10355）

阅读并分析了 GEM 论文的数据合成流程，包含 5 个核心阶段：

| Stage | 名称 | 功能 | 输出 |
|-------|------|------|------|
| Stage 1 | Tag Annotation | 筛选多步骤文本，提取元数据 | `<multi_step>True/False</multi_step>` |
| Stage 2 | Workflow & Tool Discovery | 从文本提取工作流和工具定义 | `<workflow>` + `<tools>` |
| Stage 3 | Trajectory Generation | 生成多轮对话轨迹 | `<system>/<user>/<assistant>/<tool>` 轨迹 |
| Stage 4 | Trajectory Refinement | 优化轨迹复杂度 | 更复杂的轨迹 |
| Stage 5 | Hallucination Detection | 验证轨迹质量 | 0/1 评分 |

**关键发现：**
- Stage 1 的 Tag Annotation 输出仅用于筛选，不注入后续阶段
- Stage 3 的输入是 `{candidate_tools}`（来自 Stage 2 的 `<tools>`）+ `{current_task}`（原始文本 + workflow）
- Stage 4 的输入是 Stage 3 生成的轨迹 + `{candidate_tools}`

### 2. 实现 TrajectoryRuleValidator operator

**位置：** `dataflow-extensions/dataflow_extensions/operators/agentic/trajectory_validator.py`

**验证内容：**
1. **工具定义验证**（OpenAI Schema 格式）
   - 必需字段：name, description, inputSchema
   - inputSchema 必需字段：type, properties, required
   - 工具名格式和唯一性
   - 参数类型有效性

2. **工具调用验证**
   - 工具名必须在工具集中存在
   - 所有必需参数必须提供
   - 参数类型必须匹配定义
   - 不能包含未定义的参数

3. **对话格式验证**
   - 第一条消息必须是 system
   - Turn order 正确性（system → user → assistant → tool → assistant）

### 4. 数据流总结

```
Stage 2 输出
    ↓
[<workflow>, <tools>]
    ↓
Stage 3: Trajectory Generation
    输入：{candidate_tools} ← <tools>
    输入：{current_task} ← 原始文本 + <workflow>
    ↓
输出：trajectory (JSON)
    ↓
Stage 4: Trajectory Refinement  
    输入：{trajectory} ← Stage 3 输出
    输入：{candidate_tools} ← Stage 2 输出
    ↓
输出：refined_trajectory (JSON)
    ↓
[可选] TrajectoryRuleValidator
    ↓
验证通过的轨迹
```

### 输出格式变化

论文原始格式是 XML 标签，改为 JSON 便于后续验证：

```json
{
  "trajectory": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "...", "tool_calls": [...]},
    {"role": "tool", "tool_call_id": "...", "name": "...", "content": "..."}
  ]
}
```
