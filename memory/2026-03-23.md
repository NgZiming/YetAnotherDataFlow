## 下午 15:05-15:39：重构进度存储接口（plugin.py + Pipeline.py）

### plugin.py 核心改动

**新增类型定义**：
```python
class PartitionOrBatchProgress(TypedDict, total=False):
    id: int                      # partition/batch 唯一标识
    completed_steps: List[int]   # 已完成的步骤 ID 列表
    current_steps: List[int]     # 当前正在执行的步骤 ID 列表（支持并发）
    steps_rows_nums: List[int]   # 每个步骤的数据行数（用于统计留存率）

class ProgressInfo(TypedDict, total=False):
    shard_type: str              # 'partition' | 'batch'
    partitions: List[PartitionOrBatchProgress]
    total_shards: int
    overall_status: Optional[str]  # 'running' | 'completed' | 'failed'
    start_time/last_update: Optional[str]
    error_message: Optional[str]
    extra: Optional[dict]
```

**关键设计**：
- 每个 partition 独立进度，可在不同 operator 阶段
- `current_steps` 是列表（支持一个 partition 多个步骤并发执行）
- `steps_rows_nums` 与 `completed_steps` 对应，用于计算数据留存率
- 状态统一由 `ProgressInfo` 管理，不在单个 partition 级别

**存储接口**：
- `record_progress(progress: ProgressInfo)`
- `get_progress() -> ProgressInfo`

## 技术要点总结

### 进度存储设计
- **问题**：原有 `record_steps(part, step, part_size)` 无法支持复杂进度
- **解决**：用 JSON 存储完整的 `ProgressInfo`，每个 partition 独立进度
- **恢复**：从 `completed_steps` 继续，`current_steps` 的任务可能需要重试

### YAML 安全写入
- **问题**：heredoc 遇到 `EOF` 或特殊字符会提前终止
- **解决**：base64 编码写入，`echo "{b64}" | base64 -d > file`

### 并发进度跟踪
- `current_steps` 是列表，支持一个 partition 多个步骤并发
- 提交时添加，完成后移除
